According to a document from **2025-12-18**, this reconciliation is already **enterprise-ish scale** (e.g., **586 Farm shadows**), so doing “recompute everything + build every list + maybe call AOD” on every modal open is guaranteed to feel slow.

Here’s the tuning plan I’d implement (ordered by impact / effort):

## 1) Stop recomputing `compute_expected_block(snapshot)` on modal open

**Best fix:** treat “expected” as a snapshot artifact, not a view artifact.

* When the snapshot is created, persist:

  * `expected_block` (or `__expected__`) **and**
  * a `rules_version` hash (git SHA or semver for rules)
* In `/analysis`, use stored `expected_block` **iff** `expected.rules_version == current_rules_version`.
* Only recompute when rules changed, and if you do, **write back** the new expected + version.

This alone removes the biggest O(N assets + observations) hot loop.

## 2) Cache the expensive analysis result (with invalidation)

Add columns on reconciliation row:

* `analysis_json`
* `analysis_rules_version`
* `analysis_computed_at`
* `analysis_duration_ms` (optional but useful)

Flow:

* If cached and version matches → return instantly.
* Else compute once, store, return.

You can keep it simple (DB cache) before you ever reach for Redis.

## 3) Make the analysis endpoint “summary-first” and lazy-load the heavy lists

Right now `build_reconciliation_analysis()` is doing *everything* (rollups + matched/missed/FP lists + reason codes).

Split into:

* `GET /analysis/summary` → counts, domain rollups, top-level verdict
* `GET /analysis/missed?limit=50&offset=0`
* `GET /analysis/false-positives?...`
* `GET /analysis/matched?...`

The modal can render immediately with summary, and only fetch lists when the user clicks tabs / expands sections.

## 4) Make `call_aod_explain_nonflag()` strictly on-demand

Do **not** call AOD explain while building the missed list.

Instead:

* return missed assets with a boolean `explain_available: true`
* add endpoint: `GET /analysis/missed/{asset_key}/explain`
* UI calls it only when user expands an item

This eliminates “N missed assets → N external calls” bursts.

## 5) Cut JSON load + parse cost

If snapshot JSON is large:

* **Don’t fetch the full snapshot blob** for `/analysis` unless you must.
* Prefer storing a **thin derived snapshot** for reconciliation (what analysis needs):

  * `asset_index` (canonical_key → minimal evidence flags, normalized domain/vendor, source counts, timestamps)
  * precomputed domain map
* If you keep it in Postgres, consider:

  * `jsonb` storage (if not already)
  * compressing at app level for very large payloads (or move snapshot blob to object storage and keep only pointers + derived indexes in DB)

## 6) UI-only improvement (cheap but worth doing anyway)

Even after server fixes, keep a skeleton:

* open modal instantly
* show summary skeleton
* stream in details (or progressive fetch)
