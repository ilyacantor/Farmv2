# Context & Objective
We are refactoring the core logic of AOD (AutonomOS Discover) to move away from hardcoded business rules (e.g., "if spend > $200") to a **Configuration-Driven Policy Engine**.

Currently, admission and classification logic is scattered across various ingestion and findings services. We need to centralize this into a single "Source of Truth" that controls how assets are admitted, classified (Shadow/Zombie/Clean), and flagged for risks.

**Your Goal:** Review the new Unified Policy below and prepare a technical implementation plan to refactor AOD to use a hot-reloadable `PolicyConfig` object.

# The Unified Policy (The "Constitution")

## 1. The Configuration Schema
We need a central JSON object (`PolicyConfig`) that drives all logic:
```json
{
  "admission": {
    "minimum_spend": 200,       // Threshold for Finance Gate
    "noise_floor": 2,           // Minimum sources for Shadow Gate
    "zombie_window_days": 90    // Inactivity window
  },
  "scope": {
    "include_infra": false,     // Kill List toggle (docker.com, etc.)
    "treat_directory_as_idp": false // Startup maturity mode
  },
  "exclusions": ["google.com", "microsoft.com"] // The Mutable Kill List
}

```

## 2. The Logic Flow

Logic must be applied in this strict order via a "Pure Function" (no DB side effects inside the check):

1. **The Kill List:** If domain is in `config.exclusions` OR (is infra & `!config.scope.include_infra`) -> **REJECT**.
2. **Admission Gates (OR Logic):** Admit if:
* **Finance Gate:** `spend >= config.admission.minimum_spend`
* **Governance Gate:** `in_idp` OR `in_cmdb` OR `in_cloud` OR (`config.scope.treat_directory_as_idp` && `in_directory`)
* **Shadow Gate:** `source_count >= config.admission.noise_floor`


3. **Classification (Mutually Exclusive):**
* **SHADOW:** Active + Ungoverned (No IdP/CMDB).
* **ZOMBIE:** Inactive (`last_seen > config.admission.zombie_window_days`) + Governed.
* **CLEAN:** Active + Governed.



# Implementation Requirements

1. **Central Type Definition:** Create `src/core/policy/types.ts` defining the schema above.
2. **API Exposure:** We need an endpoint `GET /api/v1/policy/config` so the Farm simulator can read these rules.
3. **Refactoring Pattern:** Use the "Strangler Pattern." Create a centralized `PolicyEngine` service. Do not rewrite the entire ingestion pipeline immediately. Instead, find the specific lines where we currently determine admission/findings and swap them to call `PolicyEngine.evaluate(asset, config)`.
4. **Lenses:** Ensure our data model supports the "Expanded Lenses" (e.g., verifying if an IdP match is a User Consent Grant vs. Managed App).

# Deliverable

Please analyze the current codebase and provide a **Step-by-Step Refactoring Plan**.

* Identify where the "old twisted logic" currently lives.
* Define the interface for the new `PolicyEngine`.
* Explain how you will expose the Config API.
* Confirm how you will handle the "Kill List" (default seeds vs. customer overrides).
