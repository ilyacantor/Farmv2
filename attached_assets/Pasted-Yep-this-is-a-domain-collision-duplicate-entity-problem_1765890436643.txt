Yep — this is a **domain-collision / duplicate-entity** problem, and it will keep biting you until you make “domain” a first-class concept.

### What’s happening

You have **3 entities** that all resolve to the same canonical key (`microsoft.com`). One of them is governed (IdP matched), two are not. So:

* **Farm** (grading at domain key) says: “microsoft.com HAS_IDP → not shadow”
* **AOD** (classifying per-entity) says: “these two entities are NO_IDP → shadow”
* Reconciliation gets “spooky” because you’re mixing **entity-level** decisions with a **domain-level** key.

### Best fix (simple + safe)

**Option C: Post-merge by domain — but only for governance + reporting.**

Don’t physically merge entities everywhere. Instead add a **domain roll-up** step:

#### Domain roll-up rule

For a given `domain_key` (registered domain):

* `domain.has_idp = OR(entities.has_idp)`
* `domain.has_cmdb = OR(entities.has_cmdb)`
* `domain.has_finance = OR(entities.has_finance)`
* `domain.has_cloud = OR(entities.has_cloud)`
* `domain.has_discovery = OR(entities.has_discovery)`
* `domain.recent_activity = MAX(entities.recent_activity)`

Then classify **shadow/zombie at the domain level** for reconciliation + UI counts.

This instantly fixes Microsoft without risking bad merges for Google/AWS/etc.

### Why not “domain-first merging” (Option A)

Because it can incorrectly merge legitimately distinct services under one domain and erase useful distinctions. You *do* want separation sometimes — just not for the *governance* outcome and reconciliation key.

### Why not “strip underscores/hyphens” only (Option B)

It’ll help, but it’s not enough. Duplicates will still happen (aliases, localized names, product variants, typos). Domain roll-up is the durable layer.

---

## UI / reconciliation implication (important)

Once you do domain roll-up:

* Counts should be **domain-counts**, not entity-counts.
* The UI should show a **“Duplicates” badge** when a domain has >1 entity variant:

Example:

> microsoft.com — Governed (IdP) ✅
> **3 variants:** “Microsoft 365”, “Microsoft_365”, “Microsoft-365”

And for transparency, you can show which variant(s) were unmatched.

---

## One crisp invariant to adopt

> **Reconciliation and “shadow/zombie” are domain-level decisions.**
> **Entities are evidence + aliases underneath that domain.**

If you keep that invariant, the spooky mismatches collapse dramatically.
