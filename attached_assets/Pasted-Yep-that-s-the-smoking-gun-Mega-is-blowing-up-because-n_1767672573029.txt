Yep — that’s the smoking gun. **Mega is blowing up because network plane scales “mult” twice**: once in number of items (implicitly, via more assets/apps/domains) and again in records-per-item. That’s not “Mega”, that’s **runaway**.

Here’s the direction that makes sense, and it’s not complicated.

## What we do (CTO call)

### 1) Fix the scaling law: per-item counts must NOT multiply with scale

At scale:

* **#items** grows with scale (more apps/assets/domains)
* **records per item** should be **constant** or **sublinear**, not multiplied

So: **remove `* mult` from per-item counts** (or replace with a capped/log scaling).

This is the correct fix. It preserves realism and makes “Mega” meaningful.

### 2) Add hard caps (budget) per plane, especially network

Even with sane scaling, we need a guardrail. The generator must obey a plane budget like:

* `NETWORK_MAX_DNS_RECORDS`
* `NETWORK_MAX_PROXY_RECORDS`
* `NETWORK_MAX_TOTAL_RECORDS`
* `NETWORK_MAX_BYTES_ESTIMATE` (optional)

If a run exceeds budget:

* either throttle further generation deterministically, or
* fail fast with a crisp error

No more accidental 200MB network blobs.

### 3) Add “Mega Lite” only if we need it, but it’s not the primary fix

“Mega Lite” is fine as a **mode** for infra testing (or UI performance), but it’s not a substitute for correct scaling.

## The actual scaling recipe (what to implement)

Replace this:

```py
num_queries = randint(5, 20) * mult
num_proxy   = randint(10, 30) * mult
```

With one of these (ranked):

### Option A (best): constant per-item + global cap

* `num_queries = randint(5, 20)`
* `num_proxy = randint(10, 30)`
* enforce global caps across the whole plane

This keeps “shape” realistic and makes total volume scale naturally with #items.

### Option B: sublinear per-item (log or sqrt) + caps

If you *really* want per-item intensity to rise with scale:

* `scale_factor = ceil(sqrt(mult))` or `ceil(log2(mult+1))`
* `num_queries = randint(5, 20) * scale_factor`
* still enforce global caps

For `mult=100`, `sqrt` gives 10 (instead of 100). Much safer.

### Option C: proportional total volume allocation

Decide total DNS/proxy budgets from scale, then distribute across items:

* total_dns_budget = f(scale)
* per item allocation = budget / num_items (with min/max clamps)

This is the most “engineered” and deterministic. Also best for keeping plane size predictable.
