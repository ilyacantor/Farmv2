1) Orchestration runtime (the actual “engine”)

Stateful execution model (not just stateless turns): a run has durable state, can pause/resume, and can survive crashes/redeploys (“durable execution” / long-running workflows).

Graph / workflow primitives: DAG + cyclic graphs (loops), conditional routing, subflows, parallel branches, joins, “map/reduce” patterns.

Interrupts + checkpoints: pause at any node, inspect/modify state, then continue (also used for approvals).

Retries, idempotency keys, compensation (Saga patterns): critical for real tool execution (payments, tickets, provisioning).

Deterministic replay: capture inputs/tool results and re-run the exact execution (or “re-run with new model/prompt”) for debugging + audits.

2) Multi-agent coordination (beyond “agent-force”)

Explicit delegation model: parent agent routes tasks to specialist agents and merges results (planner/worker patterns, committee patterns).

Shared task graph + shared memory contracts: agents don’t just “chat”; they publish structured outputs into a shared state object.

Agent-to-agent protocol support (interoperability): e.g., Microsoft has discussed/openly referenced open agent connection protocols (A2A) alongside multi-agent orchestration.

Conflict resolution: when agents disagree (tool outputs differ, policies conflict), platform has adjudication rules (vote, ranker, human escalation).

3) Tooling + integration layer (real enterprise plumbing)

Tool registry with schemas, auth requirements, rate limits, cost estimates, and safety classification.

First-class “action groups” / tool bundles (so you can grant a capability set, not 200 individual tools). Microsoft explicitly talks about curated “Action Groups” to streamline agent configuration.

Native support for tool servers / standards: MCP is now a real interoperability backbone—standardized tool exposure + invocation.

Non-API automation (“computer use”) option for last-mile apps (RPA-like web/desktop interaction) where no API exists.

Sandbox execution for risky tools (file ops, code exec), plus network egress controls.

4) Memory and context architecture (not “one blob of RAG”)

Short-term working memory vs long-term memory separation (session state vs durable user/org memory). LangGraph and Vertex AI Agent Engine both emphasize this split.

Context layering + budgets: static/org, user, session, cached, tool summaries, etc. (reduce token waste; make “what the model saw” inspectable). Vertex AI has explicitly described configurable context layers and token controls.

Memory governance: what can be written to long-term memory, retention policies, PII redaction, right-to-forget.

5) Policy, guardrails, and trust (where enterprise platforms win)

Policy engine as a first-class runtime gate (pre-tool, post-tool, pre-response): allow/deny, require approval, require extra verification steps.

Prompt-injection defenses / content & tool safety checks (input scanning, output scanning, tool argument validation). Salesforce and others frame this as “trust layer” style protection.

Human-in-the-loop controls: configurable approval points and escalation (not ad-hoc “ask the user”). This is a core pattern in agent frameworks and enterprise builders.

Separation of duties: builders vs approvers vs operators, with audit trails.

6) Observability, debugging, and auditability (must be built-in)

Full traces: every model call, tool call, latency, tokens, errors, intermediate state transitions.

Time-travel debugging: inspect prior states/checkpoints; replay from a checkpoint with modified config.

Explainability artifacts: “why did the router pick this agent/tool?” with structured rationale (not hidden chain-of-thought—think decision metadata).

Audit logs: who approved what, which data sources were accessed, what actions were taken.

(Managed agent platforms increasingly highlight observability + evaluation as core lifecycle capabilities.)

7) Evaluation + continuous improvement (agent DevOps)

Scenario-based eval harness: golden tasks, adversarial prompts, tool-failure simulations.

Offline + online evaluation: regression tests for prompts/policies + production scorecards.

Safety evals: tool misuse, data exfil attempts, prompt injection success rate.

Automatic prompt/policy versioning with rollbacks, canaries, and A/B runs.

8) Deployment and operations (the boring stuff that decides if it works)

Multi-tenant runtime with isolation boundaries (org, workspace, environment).

Versioned releases of agents, tools, policies, and memory schemas (dev/stage/prod).

Scaling + quotas: per-tenant concurrency, tool rate limits, token budgets, spend caps.

Durable queues + schedulers: timers, cron, event-driven triggers, backpressure, DLQs.

(Cloud-managed “agent engine” style services are explicitly positioning around deploy/manage/scale in production.)

9) Security, compliance, and data controls

Secret management: per-tenant vaulting, rotation, scoped credentials.

Fine-grained authorization: tool-level and object-level permissions (RBAC/ABAC), row-level filters, delegated access.

Data residency + encryption (BYOK/KMS), plus redaction pipelines.

Compliance reporting: evidence exports, access logs, retention enforcement.

10) Productization layer (what makes it a platform, not a framework)

Agent catalog / marketplace: publish, discover, install, and govern agents and tool packs.

Admin UX: policy editing, approvals, monitoring, incident management.

Developer UX: SDKs + local simulator + trace viewer + replay tooling.

Interoperability roadmap: MCP now; agent-to-agent protocols next; portable agent packaging.

Open standards momentum is real (MCP specs + ecosystem adoption, plus formal standardization efforts).
