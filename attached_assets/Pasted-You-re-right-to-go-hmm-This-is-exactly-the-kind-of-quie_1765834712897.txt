You’re right to go “hmm.” This is exactly the kind of quiet mismatch that makes the system feel haunted.

Here’s the clean CTO take:

## What’s likely true

You had **two separate truth paths** in the UI:

1. **Summary tiles**: doing its own matching with `computeMisalignment(...)` using `aod_summary.shadow_count` (an aggregate number that may not correspond to the concrete key list you diff against).

2. **Detail panel**: using backend `analysis.missed_shadows` (built with different matching rules).

So it’s totally possible to show:

* “AOD shadows: 29” (from summary)
* “Missed shadows: 5” (from analysis)
  even if AOD only returned, say, 17 keys to the reconciler.

That’s not a “bug in shadow logic.” It’s a **truth-source / matching inconsistency**.

## The fix is conceptually correct

> Use **one** reconciliation analysis result as the single source of truth for both summary and detail.

So: summary tiles should be derived from `build_reconciliation_analysis`, not from frontend recomputation + `aod_summary.shadow_count`.

## BUT (important): don’t accidentally hide ingestion issues

If AOD can return:

* a count (`shadow_count`)
* and a list (`shadow_assets[]`)
  and they can disagree, you should **surface that as its own health warning**, not silently ignore it.

### Add one invariant check (small, high value)

In the backend reconciliation output, compute:

* `aod_shadow_count_reported` (from summary)
* `aod_shadow_keys_received` (len of list)
* `aod_shadow_count_mismatch = reported != received`

If mismatch = true, show a small badge:

> “AOD payload inconsistent: count ≠ keys”

This keeps you honest and catches partial payloads / pagination / truncation.

## The other “weirdness” to watch

You mentioned “backend uses fuzzy matching.” Be careful: for reconciliation, **fuzzy matching is dangerous** unless it’s explicitly bounded and deterministic (e.g., canonical `asset_key = registered_domain`).

If fuzzy matching exists today, constrain it to:

* exact match on canonical key first
* only then fallback with a logged reason code like `MATCH_FUZZY_USED`

## What I would do next (minimal)

1. **Single source of truth** for counts = backend analysis ✅
2. **Expose payload-consistency metric** (count vs keys) ✅
3. Ensure both sides diff on **canonical key** (registered domain) and treat fuzzy as a flagged fallback ✅
