Yes — this is a good next step, and you can do it **without coupling** or “cheating”.

### The core idea

Farm can ask AOD a single neutral question:

> “For this `asset_key` in snapshot `X`, why is it **not** in your shadow (or zombie) list?”

AOD doesn’t need to know it “should have” flagged it. It just returns **its own** decision trace for **non-membership**.

---

## Minimal API shape (one endpoint, one response type)

### Farm → AOD query

`POST /reconcile/explain-nonflag`

Payload:

* `snapshot_id`
* `asset_keys: [...]`
* `ask: "shadow" | "zombie"` (or both)

### AOD → Farm response (per key)

Return only **canonical reason codes**, plus one top-level conclusion:

* `present_in_aod: true|false` (did AOD even create a candidate/asset for it)
* `decision: "not_admitted" | "admitted_not_shadow" | "admitted_not_zombie" | "unknown_key"`
* `reason_codes: [...]` (the same enum you already use)
* optional: `missing_inputs: [...]` (also enum, not prose)

That’s it. No evidence dumps. No prose.

---

## What AOD can answer even when it didn’t flag

AOD can deterministically explain *why not* via one of these buckets:

1. **UNKNOWN_KEY**

* AOD never saw it / couldn’t form candidate
  Reason codes: `NO_CANDIDATE`, `NO_EVIDENCE_INGESTED`

2. **NOT_ADMITTED**

* Saw it, but no admission gate satisfied
  Reason codes: `REJECTED_NO_GATE`, `INSUFFICIENT_DISCOVERY_SOURCES`, `STALE_ACTIVITY`

3. **ADMITTED_NOT_SHADOW**

* Admitted, but fails shadow conditions
  Reason codes: `HAS_IDP`, `HAS_CMDB`, `NO_PRESENCE_EVIDENCE`, `STALE_ACTIVITY`

4. **ADMITTED_NOT_ZOMBIE**

* Admitted, but not stale / still active / etc.

This gives Farm what it needs for reconciliation UI.

---

## The critical guardrail (prevents coupling drift)

> **AOD must not accept or consume Farm’s expected reason codes.**
> Farm only sends keys + snapshot_id + ask-type.

AOD’s explanation is based solely on **what AOD ingested and computed**.

---

## How Farm uses it in reconciliation

For each “missed by AOD” item:

* Show Farm expected reasons (already done)
* Show AOD non-flag explanation (from endpoint)
* Run the RCA reducer on the two reason-code sets

Now you have actual RCA even when AOD didn’t emit anything originally.

---

## Prompts (short, aligned)

### Prompt: Farm agent

* Compute diffs (missed/extra)
* For `missed_by_aod`, call AOD `explain-nonflag` with those keys
* Display side-by-side:

  * Farm expected reason codes
  * AOD explain-nonflag decision + reason codes
  * RCA label
* **DO NOT change code** in response to mismatches; only present RCA.

### Prompt: AOD agent

* Implement/serve `explain-nonflag` that returns decision + reason codes for each requested key
* Must work even when key is not shadow/zombie (that’s the point)
* **DO NOT** read Farm expected data
* **DO NOT change classification/admission logic** based on this; it’s diagnostic only.

---

If you want the tightest possible version of this: make `reason_codes` include a required one that always explains the *primary* reason (e.g., `PRIMARY:REJECTED_NO_GATE`) so Farm UI can show one line without drowning in tags.
