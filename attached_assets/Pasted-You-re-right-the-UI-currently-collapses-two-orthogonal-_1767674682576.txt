You’re right — **the UI currently collapses two orthogonal concepts into something that feels redundant and confusing**. This is a UX semantics problem, not a styling one.

Here’s the clean fix.

---

## Root cause (why it feels wrong)

Right now the user sees:

* **Realism** (Typical / Clean / Messy-ish)
* **Data Preset (AOD challenge level)**

  * Clean Baseline
  * Enterprise Mess
  * Adversarial

From the UI, both feel like “how messy is the data?”
But architecturally they are very different:

* **Realism = how enterprises behave**
* **Preset = how hard we’re testing AOD**

That distinction is not visible.

---

## Correct mental model (what the UI must teach)

### Axis 1 — *Enterprise Behavior*

> “What kind of company is this?”

This affects **data completeness, hygiene, maturity**.

Examples:

* Clean / Modern SaaS
* Typical Enterprise
* Messy / Legacy / M&A-heavy

### Axis 2 — *Test Intent*

> “How hard are we trying to break AOD?”

This affects **pathologies, ambiguity, edge cases**.

Examples:

* Baseline correctness
* Enterprise noise
* Adversarial ambiguity

These **must not look like siblings** in the UI.

---

## Proposed UI revision (minimal but correct)

### 1. Rename + regroup (most important change)

#### Replace this:

```
Realism
Data Preset (AOD challenge level)
```

#### With this:

### **Enterprise Behavior**

*(How this company operates)*

* Clean / Modern SaaS
* Typical Enterprise
* Messy / Legacy / M&A

> Subtext:
> “Controls data coverage, hygiene, ownership, and operational discipline.”

---

### **Test Mode**

*(How hard we challenge AOD)*

* Baseline (sanity check, deterministic)
* Noisy Enterprise (junk, gaps, partial signals)
* Adversarial (near-collisions, conflicts, ambiguity)

> Subtext:
> “Controls injected pathologies independent of enterprise maturity.”

This alone resolves 80% of the confusion.

---

## 2. Make interaction explicit (very small affordance, big clarity)

When **Adversarial** is selected, show a lightweight inline note:

> ⚠️ *Adversarial injects hard edge cases on top of the selected enterprise behavior.*

This reinforces **composition**, not replacement.

---

## 3. Optional but powerful: matrix hint (no new controls)

Under the selectors, add a single-line summary:

> **Scenario:**
> *Typical Enterprise + Adversarial* → “Realistic company with intentionally ambiguous signals”

This gives users confidence they understand what they’re generating.

---

## 4. What NOT to do

* ❌ Don’t merge Realism + Preset into one dropdown
* ❌ Don’t add more presets
* ❌ Don’t rely on tooltips alone

This is about **conceptual separation**, not verbosity.

---

## Sanity check (why this matches Farm + AOD architecture)

* Farm already treats realism and adversarial noise differently ✔
* AOD should survive adversarial patterns at *any* realism level ✔
* IRL enterprises can be clean **and** adversarial (M&A, shadow IT) ✔

So the UI should reflect that truth.

---

## One-sentence design principle (pin this)

> **Enterprise Behavior defines the world.
> Test Mode defines how hard we try to break the system.**
