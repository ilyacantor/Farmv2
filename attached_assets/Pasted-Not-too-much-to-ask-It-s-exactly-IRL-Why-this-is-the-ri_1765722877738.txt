Not too much to ask. It’s exactly IRL.

### Why this is the right IRL model

In real enterprises, users don’t type tenant IDs or snapshot IDs. The UI shows **available inventories** (tenants/environments/snapshots), you pick one, you run, and the result is recorded back to the system-of-record. The “mismatch” problem only exists because we’re letting humans hand-type identity.

### What to do next (one iteration)

Make **Farm the source of truth for “what can be run”** and make **AOD only runnable from Farm-selected snapshots**. Then AOD **pushes a reconciliation payload back to Farm automatically** when the run finishes.

That eliminates:

* tenant_id mismatch
* snapshot_id mismatch
* manual reconcile
* hidden panels

## Single Replit prompt (copy/paste to AOD agent)

> Implement IRL-grade “select source → run → automatically appears in Farm” workflow. No manual IDs.
>
> Goal:
>
> * In AOD UI, user selects **Tenant** and **Snapshot** from dropdowns populated from Farm.
> * User clicks Run.
> * When the run completes, AOD automatically posts reconciliation payload back to Farm so Farm immediately shows PASS/WARN/FAIL for that snapshot.
> * User never types tenant_id, snapshot_id, or pastes results.
>
> Requirements:
>
> 1. AOD UI dropdowns must be sourced from Farm (Farm is the source of truth):
>
> * Tenant dropdown populated from `GET /api/farm/tenants` (proxy to Farm if needed; if Farm lacks, derive from snapshots list).
>
> * Snapshot dropdown populated from `GET /api/farm/snapshots?tenant_id=...`
>
> * Snapshot selection must provide the exact `snapshot_id` that Farm uses.
>
> * Remove free-text tenant/snapshot inputs (keep an “Advanced” override only if absolutely necessary).
>
> 2. AOD run endpoint must accept only Farm-selected IDs:
>
> * `POST /api/runs/from-farm` with `{ tenant_id, snapshot_id }`
>
> * Fetch snapshot from Farm using `FARM_URL` env var.
>
> 3. After run completes, AOD must automatically sync results to Farm:
>
> * Add env var `FARM_SHARED_SECRET` (optional) and include header `X-FARM-SECRET` if set.
>
> * Call Farm endpoint `POST {FARM_URL}/api/reconcile` (or `POST /api/reconcile/auto-ingest` if that exists) with:
>
>   * snapshot_id, tenant_id, aod_run_id
>   * aod_summary (assets/findings/artifacts/zombies/shadows)
>   * aod_lists (zombie_assets, shadow_assets, top_findings) capped + deterministically ordered
>
> * If Farm is unreachable, store a “sync_pending” flag on the AOD run and show a visible warning in UI: “Run completed but Farm sync failed.”
>
> 4. Update AOD UI to show Farm sync status per run:
>
> * SYNCED / PENDING / FAILED (with one-line reason).
>
> 5. Tests:
>
> * Dropdowns load snapshots (mock Farm)
> * Running a snapshot triggers a reconciliation POST to Farm (mock Farm)
> * Farm failure yields sync_failed but run still completes
>
> Deliver proof:
>
> * Show AOD UI with tenant + snapshot dropdown (no typing).
> * Run a snapshot and show console/API proof that Farm `/api/reconcile` was called.
> * Confirm Farm UI shows reconciliation status for that snapshot without manual input.

This is the “IRL, user doesn’t think about IDs” workflow.
