# Farm Agent Prompt: Remove Self-Directed Generation, Make NLQ Tab Read-Only

## Context

Farm has two data generation paths:
- **Path A (Self-Directed):** The "Generate Ground Truth" button on the NLQ tab. Uses hardcoded defaults (`AcmeCorp-Demo`, slug `pipe_ids` like `sf_accounts`). Pushes to DCL with pipe names DCL doesn't recognize. Every push gets 422 `NO_MATCHING_PIPE`. This path is broken by design — it doesn't participate in the AAM architecture.
- **Path B (AAM-Driven):** `POST /api/farm/manifest-intake`. Receives manifests from AAM with real UUID `pipe_ids`, real `snapshot_name`, real `tenant_id`. Generates data, pushes to DCL. This is the production path and it works.

Path A is causing confusion. The operator keeps landing on the Generate button when they mean to use the AAM-driven flow. It needs to go.

## What To Do

### 1. Delete the Self-Directed Generation Path

Remove completely:
- The "Generate Ground Truth" button and its UI form (snapshot name dropdown, seed, tiers, base revenue, growth rate, quarters, push-to-DCL checkbox)
- `POST /api/business-data/generate` endpoint
- Any code that builds slug `pipe_ids` (`sf_accounts`, `ns-erp-001-invoices`, etc.) for DCL push
- The self-directed push logic in `BusinessDataOrchestrator` that sends data to DCL using Farm's internal pipe names

### 2. Keep the Ground Truth Computation

Do NOT delete the financial model engine or ground truth manifest generation logic. Specifically keep:
- The financial model builder (12 quarters of ARR, growth, churn, customer counts)
- The 8 source system generators (Salesforce, NetSuite, Chargebee, Workday, Zendesk, Jira, Datadog, AWS Cost)
- The ground truth manifest v2.0 computation (per-quarter breakdowns of revenue, CAGR, customer counts)
- Manifest validation logic

Wire this ground truth computation INTO the manifest-intake flow. When AAM dispatches a manifest and Farm generates + pushes data, Farm should also compute the ground truth from that generated data and include it in the push to DCL, tagged with AAM's `run_id` and `pipe_ids`.

### 3. Refactor the NLQ Tab to Read-Only Dashboard

The NLQ tab should display results of the most recent manifest-driven run(s). No generation triggers. No buttons that create data.

**Display:**
- **Provenance Chain:** `farm_run_id`, `dcl_run_id`, `dispatch_id`, `snapshot_name` — all sourced from AAM's manifest, not hardcoded defaults
- **Pipe IDs:** Show each UUID `pipe_id` from AAM with green/red/amber status based on DCL push result
- **Source System Cards:** Record counts per system (Salesforce: X records, NetSuite: Y records, etc.) from the manifest-driven generation
- **Ground Truth Manifest:** The computed ground truth from the last successful manifest-driven run
- **Run History:** List of recent manifest-driven runs with timestamps, status, pipe counts

**Behavior:**
- On page load, fetch the latest manifest-driven run results from Farm's DB
- A "Refresh" button that re-fetches — does NOT regenerate data
- If no manifest-driven runs exist yet, show a clear message: "No AAM-dispatched runs found. Run the AAM pipeline to generate data."
- No dropdowns, no seed inputs, no "Generate" buttons

### 4. Clean Up API Surface

After removing the self-directed path:
- `POST /api/farm/manifest-intake` — keep (single manifest from AAM)
- `POST /api/farm/manifest-intake/batch` — keep (batch manifests from AAM)
- `POST /api/business-data/generate` — DELETE
- `GET /api/farm/ground-truth/{run_id}` — keep, but ensure it returns ground truth from manifest-driven runs
- Any endpoint that only served the self-directed path — DELETE

### 5. Update Tests

- Delete tests for self-directed generation and push
- Keep tests for manifest-intake flow
- Add tests confirming ground truth is computed and included during manifest-driven execution
- Ensure zero regressions on the 74 existing business data tests that test the generators themselves (not the push logic)

## What NOT To Do

- Do NOT delete the source system generators. They are used by both paths — the manifest-intake path routes to them by `source.category`.
- Do NOT delete ground truth computation. Just move it into the manifest-driven flow.
- Do NOT add any new buttons or generation triggers to the NLQ tab.
- Do NOT create fallback behavior. If no manifest runs exist, say so. Don't generate demo data.
- Do NOT change the manifest-intake API contract. AAM depends on it.

## Success Criteria

1. The NLQ tab has zero generation controls — no buttons, no forms, no dropdowns that trigger data creation
2. The NLQ tab displays real provenance from AAM-driven runs with UUID `pipe_ids`
3. Ground truth is computed automatically as part of manifest-intake processing
4. `POST /api/business-data/generate` returns 404
5. All manifest-intake tests still pass
6. The operator's workflow is: run AAM → open Farm NLQ tab to verify → go to DCL. Zero clicks in Farm.
